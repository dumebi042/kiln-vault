5.1.1
Rental / User info can be overwritten during its rental period
Severity: Critical Risk
Context:
• vNFT.sol#L327
• vNFT.sol#L356
• vFactory.sol#L595
• vFactory.sol#L808
• MinimalRecipient.sol#L48
Description: Rental / User info can be overwritten during its rental period. This can be achieved due to the fact that
when the withdrawal recipient of a token's validator has a positive balance during the setUser or clearUsership
call, we would get a call back to the current
_beneficiaryOf(tokenId) or userInfo.user and then after this
callback we would actually set the new user info in storage. So a malicious owner or a current user of a token can
take advantage of this callback to set the user info multiple times by reentring into these endpoints.

called.

5.2.7
vNFT - The freeze mechanism does not fully mitigate front-running related issues
Severity: High Risk
Context: vNFT.sol#L36
Description: The vNFT contract is tokenizing users' deposits into validators, i.e., users calling purchase deposit eth
through the vFactory which is allocated to validators, and in return they are given an NFT in the vNFT contract to
represent their ownership. This NFT is transferable and thus, can be traded on decentralized exchanges. Malicious
actors can take advantage of the fact that the tokenId is not tied to changes in the token storage state. A classic
example would be a malicious vNFT seller that can potentially front-run the actual dex swap transaction in the mem-
pool, causing the buyer to purchase a depreciated token, but there could be many different attack vectors, some
of which will be described later in this issue. The current version of the code implements the freezing mechanism
to mitigate this specific issue. In short, the freezing mechanism disallows transfers of vNFT tokens for a certain
amount of time after setUser and withdraw were called and for an infinite amount of time after exit was called.
The idea behind this design is to revert the immediate swap transaction, giving the other party time to investigate
the error and cancel the deal. In practice, however, this solution is far from fully mitigating all front-running related
issues for a myriad of reasons, namely:
1. Many users won't be able to understand that the transaction failed due to IllegalTransferWhileFrozen
since it is not supported in 3rd party decentralized exchanges and thus may cause them to try and buy it
again after the freeze period ended, opening the possibility for an attack once the initial freezing period is
over.
12


--- Page 14 ---
2. Some NFT decentralized exchanges implement auction methods and contracts that differ from the classic
"seller as a maker, buyer as a taker", although less common, it is possible that the buyer would be the maker,
signing a buy order off-chain that is supposed to be submitted later by the seller as part of a swap transaction.
In case the expiration of this buy order is longer than the freeze period, the seller might first call withdraw
or/and setUser, wait until the freezing period is over, and then submit the swap transaction, and by doing
this, selling a depreciated token to the buyer.
Recommendation: As already mentioned, the issue is caused due to the fact that the tokenId is not tied to
changes in the token storage state. We propose to implement a mechanism where there are two different identi-
fiers for any token, an internal identifier that should stay constant throughout the entire token life-cycle (it can be
implemented as a counter), and an external identifier. The internal identifier will be used to find the corresponding
storage data attached to the token. Whenever the token storage data is changed (setUser, withdraw, exit for
example), the old token should be burned, and a new token should be minted using a new external identifier.
The external identifier might be implemented as hash(internal_id, version), where the version is incremented
whenever the token storage data is changed.
Kiln: Fixed in d05f25b2d809823d05db13b8a82aa825e0d04c09 by implementing the auditor's recommendation
for setUser and withdraw, and for exit implement a different mechanism that achieves the same result.
Spearbit: Fixed.

for pool A and vice versa.

maintain another mapping that maps from the pool's address to the pool's index and then change the _set-

were included before sending the next one
